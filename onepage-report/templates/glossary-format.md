# glossary.md 格式規範（分級版）

> **設計原則**：80/20 原則 - 核心術語詳細解釋，次要術語簡短帶過

## 術語分級策略

| 級別 | 數量 | 說明長度 | 適用對象 |
|------|------|---------|---------|
| **核心術語** | 3-5 個 | 5-7 行詳細解釋 | 報告核心概念、非常見詞彙、容易誤解的術語 |
| **次要術語** | 5-10 個 | 1-2 行簡短解釋 | 常見技術術語、已知縮寫、背景知識 |

### 核心術語判定標準

符合**任一條件**即為核心術語：
1. **報告核心概念**：與報告主題直接相關的關鍵術語
2. **非常見詞彙**：一般人不熟悉的專業術語
3. **容易誤解**：看起來簡單但容易誤解的術語

## 術語分類

| 類別 | 範例 | 解釋重點 |
|------|------|----------|
| 架構術語 | cluster、L2 cache、NUMA | 是什麼、在系統中的角色 |
| 指標術語 | FPS、1% Low、Jank、latency | 代表什麼、為何重要 |
| 機制術語 | migration、scheduling、cache miss | 怎麼運作、影響什麼 |
| 縮寫 | POC、SoC、CCX | 全名、簡單解釋 |

## 範例（分級版）

```markdown
## 術語詞彙表

本報告中的專業術語解釋，適合非技術背景讀者參考。

### 核心術語

#### G1: Frame Queue
- **類別**：機制術語
- **白話解釋**：當 CPU 處理畫面的速度比 GPU 渲染快時，會產生一堆「排隊等待渲染」的畫面。這些排隊的畫面就叫 Frame Queue。問題是，等你看到這些畫面時，它們已經是「舊的」了，你的操作反應在螢幕上就會有延遲感。
- **在本報告的角色**：Anti-Lag SDK 的核心目標就是消除這個排隊現象，讓你看到的畫面盡可能是最新的。
- **類比說明**：像是餐廳廚房出菜太快，服務生來不及送，菜放久了就涼了。

#### G2: Click-to-Photon
- **類別**：指標術語
- **白話解釋**：從你手指點擊螢幕的那一刻，到螢幕上的像素真正改變，中間經過的總時間。這個時間越短，遊戲操作感越靈敏。
- **在本報告的角色**：這是我們衡量延遲改善的主要指標，POC 目標是降低 5% 以上。

#### G3: Anti-Lag SDK
- **類別**：架構術語
- **白話解釋**：一套讓遊戲引擎和處理器「握手同步」的工具。它會在適當的時機暫停 CPU，等 GPU 準備好再繼續，避免產生 Frame Queue。
- **在本報告的角色**：這是本報告提案要導入的核心技術，參考 AMD 在 PC 平台已驗證的方案。

### 次要術語

| 術語 | 類別 | 簡短解釋 |
|------|------|---------|
| FPS | 指標 | 每秒顯示幾張畫面，數字越高越流暢 |
| 1% Low | 指標 | 最差 1% 時間的 FPS，代表最卡的時候有多卡 |
| VSync | 機制 | 垂直同步，讓畫面更新與螢幕刷新對齊 |
| SurfaceFlinger | 架構 | Android 系統的畫面合成器，負責把多個 App 的畫面合成一張 |
| BufferQueue | 機制 | Android 用來暫存畫面的緩衝區，天生有 1-2 幀延遲 |
| FPSGO | 架構 | 聯發科的幀率穩定技術，但無法感知引擎內部狀態 |
| POC | 縮寫 | Proof of Concept，概念驗證實驗 |
| SoC | 縮寫 | System on Chip，系統單晶片（手機處理器） |
```

## 產生規則

1. 掃描 one_page.md 中所有可能的專業術語
2. **先判定核心 vs 次要**（參照分級標準）
3. 核心術語：用「國中生能懂」的白話文詳細解釋
4. 次要術語：用 1 行簡短解釋，放入表格
5. 術語總數：核心 3-5 個 + 次要 5-10 個 = 總計 8-15 個
