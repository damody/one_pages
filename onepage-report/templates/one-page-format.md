# one_page.md 格式規範

## 完整結構模板

```markdown
# {結論標題，要有數字或明確結論}
{副標題：靈感來源或目標說明}

## 已驗證的成功要素
{說明 PC/Server 平台或業界已驗證有效的前提條件}
- {要素 1，詳細說明} [C1]
- {要素 2，詳細說明} [C2]
- {要素 3，詳細說明} [C3]
- {已驗證的效果，含具體數字} [C4]

## 現況與問題
{說明目前平台（如 Dimensity）的現況與瓶頸}
- {現況/問題 1，詳細說明} [C5]
- {現況/問題 2，詳細說明} [C6]
- {現況/問題 3，詳細說明} [C7]

## 技術關鍵點 / 為何可能有效
{說明為何在手機平台可能有類似效果}
- {關鍵點 1，詳細說明} [C8]
- {關鍵點 2，詳細說明} [C9]
- {推論結論，詳細說明} [C10]

## 對照表（可選）
| 比較項目 | PC 平台 | 手機平台 |
|----------|---------|----------|
| {項目1}  | {PC做法} | {手機做法} |
| {項目2}  | {PC做法} | {手機做法} |

## 預期效益
- {效益 1，量化優先}
- {效益 2}
- {效益 3}

## POC 設計
實驗條件：
- A 組 (Baseline)：{現行方案}
- B 組 (Experimental)：{新方案}

遊戲場景：
- {場景 1}
- {場景 2}

## 成功判定準則
1. {判定項目 1}
   {具體指標與門檻}
2. {判定項目 2}
   {具體指標與門檻}
3. {判定項目 3}
   {具體指標與門檻}

## 行動
{決策建議，需符合報告目的，可包含時程/條件}
```

---

## 區塊彈性調整

區塊可根據內容彈性調整：
- 如果不是技術 POC 題目，可以省略「已驗證的成功要素」
- 如果沒有對照需求，可以省略「對照表」
- **核心區塊**：標題、現況/問題、證據/效益、行動

---

## 撰寫原則（國中生能懂標準）

### 基本要求

- 所有內容必須讓國中生能看懂
- **優先使用白話文**，盡可能詳細解釋
- **只有字數太多時**才改用術語，並加 `[[術語]]` 標記連結到附錄
- 術語標記會自動連結到附錄的白話解釋 + 圖解說明
- 寧可多寫幾句白話文，也不要為了精簡而用看不懂的術語

### 標題撰寫原則

- 必須是結論句，不是主題句
- 好：「導入快取可降低 80% 回應時間」
- 壞：「系統效能改善方案」

---

## 內容密度原則（MTK 風格）

- 一頁投影片要有足夠資訊量，讓主管能做決策
- 每個 bullet 都要有實質內容，不要空泛
- 數字要具體（不要「大幅改善」，要「改善 81%」）
- 可以有多個 bullet，內容量不設限制
- 要有「已驗證」→「現況問題」→「為何可行」→「POC 設計」→「成功判定」的邏輯鏈
- **內容完整性優先**：盡可能保留所有技術細節，Phase 6 會使用 Yoga Layout 自動處理排版

### 排版由 Phase 6 自動處理

- **無字數限制**：Phase 3 不再限制每個 bullet 的字數
- **自動字體縮放**：Yoga Layout 會自動計算最適字體大小
  - 本文最小 10pt
  - 小字註解最小 8pt
- **自動分欄/分頁**：內容過多時自動調整佈局

## Citation 引用格式

每個論述都應引用素材來源：

```markdown
- 降低跨 cluster 遷移可減少 cache miss [C3]
- PC 平台實測改善 40% [C1][C2]
```

引用的 ID 對應到 Citation Map 中的來源記錄。
